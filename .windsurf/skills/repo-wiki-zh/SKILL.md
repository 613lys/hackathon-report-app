# repo-wiki-zh Skill

## 输出目录结构

始终在项目根目录下创建以下固定结构：

```
wiki/
├── index.md              # 项目首页：badges、模块表、快速开始
├── architecture.md       # 系统图、技术栈、数据流、ER图、安全风险
├── getting-started.md    # 环境搭建、运行、测试、FAQ
├── doc-map.md            # 文档关系图、按角色的阅读路径
├── {领域}/               # 每个业务领域一个文件夹（如 后端/、前端/、数据库/）
│   ├── _index.md         # 领域概述 + 分层架构图
│   └── {模块}.md         # 模块详情：类图、方法详解、示例
├── api/
│   └── {服务}-api.md     # REST/RPC 接口参考
└── progress.json         # 生成进度追踪（第一个创建）
```

## 工作流

### 第一步 — 探索（开始写任何文档之前）

并行执行以下操作：
- 列出根目录结构
- 读取关键配置文件（`package.json`、`pom.xml`、`requirements.txt`、`go.mod` 等）
- 读取主入口和路由/控制器文件
- 读取数据库 schema 或迁移文件
- 读取 2-3 个核心源文件，理解代码模式

然后构建**模块列表**：按领域分组，列出每个需要生成 wiki 页面的文件。

### 第二步 — 规划批次

根据项目规模选择生成策略：

| 规模 | 模块数 | 策略 |
|------|--------|------|
| 小型 | ≤ 10 | 顺序生成所有文档 |
| 中型 | 11–30 | 批次A：index + architecture；批次B：核心模块；批次C：外围模块 + API |
| 大型 | > 30 | 批次A：仅生成 index + architecture + doc-map；其余模块列入 `progress.json`；每轮对话续写 3–5 个模块 |

首先创建 `progress.json`：
```json
{
  "scale": "medium",
  "total": 12,
  "completed": [],
  "pending": ["index", "architecture", "..."],
  "current_batch": 1
}
```

### 第三步 — 生成（逐批进行）

按以下优先级顺序生成文件：
1. `index.md`
2. `architecture.md`
3. `getting-started.md`
4. 各领域 `_index.md`
5. 模块文档（核心模块 → 外围模块）
6. `api/*.md`
7. `doc-map.md`

每个文件遵循 `references/templates.md` 中对应的模板。

### 第四步 — 交叉链接

每个文档末尾必须有**相关文档**章节，链接到：
- 父级领域 `_index.md`
- 调用它的模块或被它调用的模块
- 对应的 API 参考（如适用）

## 图表规则

每个文档至少包含 1-2 个 Mermaid 图表，按内容类型选择：

| 内容 | 图表类型 |
|------|---------|
| 模块依赖、系统架构 | `flowchart TB` |
| 类的字段和方法 | `classDiagram` |
| 请求/响应、服务间调用 | `sequenceDiagram` |
| 数据库表及关系 | `erDiagram` |
| 组件生命周期、状态转换 | `stateDiagram-v2` |
| 高层概念地图 | `mindmap` |

图表必须放在带有 ` ```mermaid ` 语言标签的代码块中。

## 模块文档结构

每个 `{模块}.md` 使用以下结构（简洁为主，优先用表格而非长段落）：

```
# {类名 / 组件名}

## 概述（最多 2-3 句话）

## 架构位置（flowchart：调用方 → 当前模块 → 依赖方）

## 类图（classDiagram）

## 方法详解

每个公开方法：
### `methodName(params)`
一句话说明。链接到源文件和行号。

```语言
// 基础用法
...
```

```语言
// 错误/边界情况
...
```

```语言
// 边缘情况或安全关注点
...
```

## 安全分析（固定包含此章节）
表格：漏洞类型 | 位置 | 严重程度 | 修复建议

## 相关文档
```

---

## 安全分析格式

**所有模块文档必须包含此章节**，即使没有发现任何问题。格式：

```markdown
## 安全分析

| ID | 类型 | 位置 | 严重程度 | 修复方案 |
|----|------|------|----------|---------|
| VUL-001 | SQL 注入 | `method()` 第42行 | 🔴 严重 | 使用参数化查询 |
| VUL-002 | 缺少校验 | `createX()` | 🟡 中等 | 添加 @NotBlank 注解 |

> ✅ 本模块未发现安全问题。（无问题时使用此行）
```

严重程度：🔴 严重 · 🟠 高危 · 🟡 中等 · 🟢 低危

## index.md 必要内容

- 项目名称 + badges（技术栈、版本、语言）
- 3 句话项目描述
- 架构预览 `flowchart`（仅顶层，链接到 architecture.md）
- 文档导航表（5-8 行）
- 模块概览表（名称 | 文件路径 | 职责）
- 快速开始（最多 3-5 条命令）

---

## architecture.md 必要内容

- 执行摘要（最多 5 句话）
- 完整系统 `flowchart`（包含所有模块及其连接）
- 技术栈表（组件 | 技术 | 版本 | 用途）
- 数据流 `sequenceDiagram`（主要用户操作流程）
- 数据库 `erDiagram`（如适用）
- 目录结构（代码块，只保留关键路径）
- 应用的设计模式（表格）
- 安全风险摘要（链接到各模块文档详情）

---

## 大型项目处理

对于模块数 > 30 的仓库：
- 跳过测试文件、生成代码、vendor/node_modules 目录
- 按优先级处理：入口文件 > 控制器 > 服务 > 工具类
- 对低优先级模块生成**存根页面**：仅概述 + 类图，无方法详解
- 每批完成后更新 `progress.json`
- 在开始时告知用户："这是一个大型项目，我将分批生成文档。第一批：index + architecture。"

## 质量检查（完成每个文件前验证）

- [ ] 至少包含 1 个 Mermaid 图表
- [ ] 安全分析章节存在
- [ ] 底部有相关文档章节
- [ ] 没有只有标题和列表的章节——每节必须有正文或表格
- [ ] 代码示例使用正确的语言标签（java、typescript、python、sql、bash 等）
- [ ] 每个方法包含源文件链接（`[📄](file://路径/文件.java#L10-L20)`）
