## Context

当前报表管理系统使用Angular框架，前端样式分散在几个文件中：
- `styles.css` - 全局基础样式（仅6行）
- `login.component.ts` - 登录页面内联样式
- `report-viewer.component.css` - 报表管理主界面样式
- `report-run-flow.component.ts` - 审批流程时间线内联样式

现有问题：
- 无统一设计系统，颜色硬编码
- 组件间样式不一致
- 缺乏响应式设计
- 交互反馈不够现代

## Goals / Non-Goals

**Goals:**
- 建立基于CSS变量的统一设计系统
- 将主色调从亮绿色（#4CAF50）改为专业蓝色（#2563eb）
- 为按钮建立视觉层次（主/次/危险）
- 优化表格、卡片、表单的视觉设计
- 添加微交互效果（悬停、聚焦、过渡动画）
- 提供基础响应式适配
- **保持所有HTML文字内容不变**

**Non-Goals:**
- 不修改Angular框架或组件逻辑
- 不修改后端API
- 不添加新功能
- 不改变页面结构或布局位置（仅优化现有布局的视觉效果）
- 不引入第三方UI组件库

## Decisions

### 1. 使用CSS变量而非SCSS/LESS
**选择：** 纯CSS变量
**理由：**
- 无需额外构建工具，与现有Angular配置兼容
- 运行时可通过JavaScript动态修改（未来扩展）
- 现代浏览器全面支持
- 避免引入编译步骤

**替代方案：** SCSS预处理器
**拒绝理由：** 需要修改构建配置，对当前优化来说过于复杂

### 2. 颜色系统：专业蓝主色调
**选择：**
- 主色：#2563eb（专业蓝）替代 #4CAF50（亮绿）
- 语义化颜色：成功(#10b981)、警告(#f59e0b)、错误(#ef4444)、信息(#3b82f6)
- 灰度系统：50-900 9级灰度

**理由：**
- 蓝色更适合企业级应用，传达专业和信任感
- 语义化颜色便于状态标识（成功/错误等）
- 灰度系统提供足够的层次区分

### 3. 阴影和圆角规范
**选择：**
- 阴影：3级（sm/md/lg），基于Material Design原则
- 圆角：6px/8px/12px/16px，逐步递增

**理由：**
- 一致的阴影层级建立深度感
- 适度的圆角软化视觉，避免过于生硬

### 4. 按钮层次设计
**选择：**
- 主按钮：蓝色填充，白色文字，悬停上浮效果
- 次按钮：浅灰背景，深灰边框，深灰文字
- 危险按钮：红色填充，用于拒绝等破坏性操作
- 禁用状态：降低透明度，移除交互效果

**理由：**
- 清晰的操作层次引导用户
- 一致性降低学习成本
- 悬停动画提供即时反馈

### 5. 仅修改样式文件，不动TypeScript逻辑
**选择：** 只修改样式部分（CSS/内联styles）
**理由：**
- 最小化风险，避免功能回归
- 快速实施，易于验证
- 符合用户"不改变文字内容"的要求

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| CSS变量在某些旧浏览器不兼容 | 项目使用现代Angular，目标浏览器已支持CSS变量；如需兼容可添加fallback |
| 样式权重问题导致覆盖失败 | 使用!important谨慎处理，确保新样式优先级足够 |
| 响应式断点选择不当 | 使用标准768px断点，覆盖大部分移动设备 |
| 视觉突变用户不适应 | 颜色从绿到蓝是显著变化，但这是设计升级的预期效果 |

## Migration Plan

**实施步骤：**
1. 更新 `styles.css` - 建立设计系统变量
2. 更新 `login.component.ts` - 登录页样式
3. 更新 `report-viewer.component.css` - 主界面样式
4. 更新 `report-run-flow.component.ts` - 时间线样式

**验证方式：**
- 运行 `npm start` 查看各页面效果
- 验证响应式布局（调整浏览器宽度）
- 检查交互效果（悬停、聚焦）

**回滚策略：**
- 所有修改在独立commit中，可直接revert
- 备份原始样式已存在于git历史
