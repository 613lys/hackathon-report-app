# 后端压测规范

## 新增需求

### 需求：并发流程被真实模拟

压测套件必须在至少 100 个并发虚拟用户下执行 Maker 登录、报表运行、提交以及 Checker 审批等流程，以暴露竞态与资源争用问题。

#### 场景：压测运行覆盖 Maker/Checker 行为

- **当** 使用默认配置执行压测任务
- **则** 套件会生成超过 100 个混合的 Maker 与 Checker 用户，持续执行登录、报表运行、提交与审批，无需串行化。

### 需求：阈值可配置且严格执行

套件应通过 Gradle/系统属性提供虚拟用户数、升压时间、失败/延迟阈值等配置，并在观测指标超出阈值时使压测任务失败。

#### 场景：延迟超标导致构建失败

- **当** 95 分位延迟超过配置的最大值
- **则** 压测任务以非零退出码终止，并在控制台输出详细指标。

### 需求：环境保持隔离

套件必须在每次迭代之间重置 H2 数据库状态与共享缓存，以避免测试数据串扰，并能在本地与 CI 环境中无人工干预地运行。

#### 场景：每轮执行后刷新数据库

- **当** 某个压测场景完成一批流程
- **则** 框架在下一批开始前回滚或重建数据库结构，保证后续迭代使用一致的种子数据。
